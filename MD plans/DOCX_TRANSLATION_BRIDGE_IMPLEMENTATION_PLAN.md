# DOCX Translation Bridge — Implementation Plan

## Purpose

Add an experimental DOCX translation bridge to the Long Document LLM Workstation.

The goal is **not** full Microsoft Word fidelity in the first version. The goal is a practical, auditable workflow:

```text
DOCX
→ extract visible text from DOCX XML
→ create segmented translation source file
→ translate segments using existing translation workflow
→ create bilingual lookup/manifest
→ patch DOCX XML by segment location
→ repackage translated DOCX
```

This feature should reuse the existing segmentation, translation, provider, and validation architecture where practical.

---

## Design Principle

The model translates text.  
The program owns document structure.

Do **not** ask the model to understand or rebuild DOCX. DOCX structure is handled by deterministic code.

The model should see controlled tagged/segmented text only.

---

## Scope

### In scope for v1

Support basic DOCX body translation from:

```text
word/document.xml
```

Handle these structures:

- normal paragraphs
- headings
- list-item paragraphs
- table-cell paragraphs
- visible text stored in `<w:t>` nodes

Preserve:

- DOCX package structure
- paragraph order
- table placement
- list placement
- paragraph styles
- basic document readability
- original source DOCX unchanged
- bilingual lookup table
- extraction/patch manifest

### Out of scope for v1

Do not support yet:

- exact inline formatting preservation
- comments
- tracked changes
- headers
- footers
- footnotes
- endnotes
- text boxes
- shapes
- SmartArt
- embedded objects
- OCR
- PDF
- password-protected DOCX
- macro-enabled DOCM
- full CAT-tool style round-trip fidelity

These can be added later after the simple bridge works.

---

## Core Warning

Do **not** use global text find/replace in `document.xml`.

Bad approach:

```text
Find source_text in XML and replace every occurrence.
```

Why this is unsafe:

- the same text may appear multiple times
- XML may split text across multiple runs
- source text may be escaped
- replacement could hit the wrong location
- duplicated strings would be corrupted
- document structure could be damaged

Correct approach:

```text
Extract text with segment IDs and XML locations.
Patch the exact XML location recorded for each segment.
```

The bilingual lookup table is for audit and QA.  
The manifest location data is for patching.

## User Workflow

### Phase 1 — Extract DOCX

User selects:

```text
input.docx
```

Program creates:

```text
docx_work/extracted_segments.txt
docx_work/manifest.json
docx_work/bilingual_lookup.csv
```

The extracted segment file should use the existing delimiter format:

```text
%%% SEGMENT 0001 START %%%
Local LLM Coding Benchmark
%%% SEGMENT 0001 END %%%

%%% SEGMENT 0002 START %%%
Purpose
%%% SEGMENT 0002 END %%%
```

### Phase 2 — Translate

User translates `extracted_segments.txt` using existing translation workflow.

Output:

```text
translated_segments.txt
```

### Phase 3 — Build bilingual lookup

Program compares:

```text
extracted_segments.txt
translated_segments.txt
manifest.json
```

and writes:

```text
bilingual_lookup.csv
```

### Phase 4 — Patch DOCX

Program unzips original DOCX, patches `word/document.xml` using the manifest locations, then rezips:

```text
translated_output.docx
```

### Phase 5 — Validate

Program verifies:

- every source segment has a matching target segment
- target segment IDs match source IDs
- no missing target segments
- no extra target segments
- XML remains parseable
- DOCX can be opened as a ZIP
- `[Content_Types].xml` and package structure remain present

## Extraction Strategy

### v1 extraction mode: paragraph mode

Extract full paragraph text by joining all visible `<w:t>` nodes under each `<w:p>`.

Why paragraph mode:

- better translation quality than translating individual runs
- preserves natural sentence context
- easier for LLMs
- fits existing segment workflow

Example:

```xml
<w:p>
  <w:r><w:t>Please review the </w:t></w:r>
  <w:r><w:t>updated informed consent form</w:t></w:r>
  <w:r><w:t> before the visit.</w:t></w:r>
</w:p>
```

Extract as one segment:

```text
Please review the updated informed consent form before the visit.
```

Do not extract each run separately unless a later advanced mode needs it.

---

## Patch Strategy

### v1 patch mode: first-run replacement

For each paragraph segment:

1. Find the paragraph by recorded paragraph index/location.
2. Find all `<w:t>` nodes in that paragraph.
3. Put the full translated paragraph into the first `<w:t>`.
4. Clear text in the remaining `<w:t>` nodes.
5. Preserve paragraph XML, paragraph style, numbering, table location, and overall structure.

This may lose some inline formatting inside the paragraph, but it preserves document layout better than plain text export.

Example:

Before:

```xml
<w:p>
  <w:r><w:t>Please review the </w:t></w:r>
  <w:r><w:t>updated informed consent form</w:t></w:r>
  <w:r><w:t> before the visit.</w:t></w:r>
</w:p>
```

After:

```xml
<w:p>
  <w:r><w:t>Prosím, skontrolujte aktualizovaný formulár informovaného súhlasu pred návštevou.</w:t></w:r>
  <w:r><w:t></w:t></w:r>
  <w:r><w:t></w:t></w:r>
</w:p>
```

Later, an advanced patcher can distribute translated text across runs or preserve inline spans more intelligently.

## Segment Format

Use the existing translation delimiter format:

```text
%%% SEGMENT 0001 START %%%
[source text]
%%% SEGMENT 0001 END %%%
```

Rules:

- segment IDs are four-digit zero-padded
- IDs are stable for the run
- do not renumber after extraction
- empty paragraphs are skipped by default
- whitespace-only paragraphs are skipped by default
- optional future setting: preserve empty paragraphs as non-translatable markers

## DOCX ZIP Handling

DOCX is a ZIP package.

Use standard library:

```python
zipfile
xml.etree.ElementTree
tempfile
shutil
pathlib
hashlib
csv
json
```

No external dependencies required for v1.

Basic unpack flow:

```text
1. Create temp working directory.
2. Extract all files from input DOCX.
3. Parse word/document.xml.
4. Extract/patch as needed.
5. Write modified document.xml.
6. Zip the full folder back into output DOCX.
```

When rezipping, preserve relative paths exactly:

```text
[Content_Types].xml
_rels/.rels
word/document.xml
word/_rels/document.xml.rels
word/styles.xml
...
```

Do not zip the parent temp folder itself. Zip its contents.

## XML Namespace Handling

WordprocessingML uses namespaces.

The main namespace:

```text
http://schemas.openxmlformats.org/wordprocessingml/2006/main
```

Use namespace map:

```python
NS = {
    "w": "http://schemas.openxmlformats.org/wordprocessingml/2006/main"
}
```

Find paragraphs:

```python
root.findall(".//w:p", NS)
```

Find text nodes inside a paragraph:

```python
paragraph.findall(".//w:t", NS)
```

When writing XML back, preserve namespaces as much as possible.

Use:

```python
ET.register_namespace("w", NS["w"])
```

Potential risk:

```text
xml.etree.ElementTree may rewrite namespace prefixes.
```

Usually Word tolerates this, but if prefix preservation becomes important later, switch to `lxml`.

Do not add `lxml` dependency in v1 unless needed.

## Text Extraction Details

Visible text is usually in:

```xml
<w:t>Text here</w:t>
```

Paragraphs can contain:

```xml
<w:tab/>
<w:br/>
<w:softHyphen/>
```

v1 handling:

- `<w:t>` text: extract
- `<w:tab/>`: convert to `\t` or space
- `<w:br/>`: convert to `\n` inside segment if practical
- other unsupported inline elements: ignore or log warning

Whitespace:

- preserve leading/trailing whitespace only if meaningful
- record original source text hash after extraction
- use `xml:space="preserve"` if translated text has leading/trailing spaces

Simple v1 rule:

```text
strip extracted paragraph text for translation segments
```

But record whether original had leading/trailing whitespace if later needed.

---

## Table Handling

Tables are inside `word/document.xml`, but table cell text is still inside paragraphs.

For v1, extracting paragraphs via `.//w:p` automatically extracts table-cell paragraphs too.

Optional manifest fields can identify table context:

```text
table_index
row_index
cell_index
```

If table context is too costly for v1, leave these as null.

Important:

- preserve table XML structure
- patch text inside paragraph location only
- do not rebuild tables manually

---

## Lists and Headings

Lists are also paragraphs with numbering properties.

For v1:

- extract list item paragraph text normally
- patch the paragraph text normally
- preserve paragraph properties `<w:pPr>`
- do not touch numbering XML

Headings:

- detect style ID from `<w:pStyle w:val="Heading1"/>`
- store `style_id` in manifest
- patch text only
- do not touch style XML

---

## Validation Before Patch

Before modifying DOCX, validate translated segments:

Required checks:

- every manifest segment ID exists in translated output
- no duplicate target segment IDs
- no extra target segment IDs unless allowed
- target text is not empty unless source was empty
- segment delimiter format is valid
- translation output does not contain obvious model commentary outside segments

Optional checks:

- protected terms
- placeholders
- numbers/dates/units
- tag preservation if tagged content is used

If validation fails:

```text
Do not patch DOCX by default.
Show/report errors.
Allow force patch only later, not in v1.
```

---

## Validation After Patch

After patching:

- parse modified `word/document.xml`
- verify segment count patched
- verify output DOCX exists
- verify output DOCX is a valid ZIP
- verify required DOCX files exist:
  - `[Content_Types].xml`
  - `_rels/.rels`
  - `word/document.xml`

Optional:

- attempt to reopen/reparse output `word/document.xml`
- write patch report

Patch report example:

```md
# DOCX Patch Report

Input: input.docx
Output: translated_output.docx
Segments extracted: 42
Segments translated: 42
Segments patched: 42
Warnings: 2

## Warnings

- Segment 0012 target is much longer than source.
- Segment 0031 had 5 original text runs; v1 patch collapsed text into first run.
```

## Error Handling

Handle these cases:

### Input errors

- input file missing
- input file is not `.docx`
- file is not a valid ZIP
- missing `word/document.xml`
- encrypted/protected file
- XML parse failure

### Extraction errors

- no text segments found
- unsupported XML structure encountered
- segment ID collision
- write permission failure

### Translation/patch errors

- translated output missing segment
- translated output has duplicate segment
- translated output has malformed delimiter
- manifest missing
- source hash mismatch
- XML location no longer exists
- output DOCX cannot be written

### Safety behavior

If patching fails:

- keep original DOCX untouched
- keep extracted segments and manifest
- write failure report
- do not produce partial DOCX unless explicitly marked as failed/partial

## Safety Rules

- Never modify the source DOCX in place.
- Always write a new DOCX.
- Always keep manifest and lookup table.
- Always validate segment IDs before patching.
- Always patch by XML location, not source text search.
- Never store full DOCX content in SQLite.
- Never send DOCX binary to model.
- Only send extracted text segments to model.
- Do not preserve or include real client/study/project sample data in tests.

## Final Ruling

The DOCX bridge is feasible and worth building, but it must be done as a controlled intermediate pipeline.

Correct MVP:

```text
DOCX → paragraph segments + manifest → translation → bilingual lookup → location-based XML patch → new DOCX
```

Avoid:

```text
global text find/replace
run-by-run translation
full DOCX fidelity promises
```

This gives a practical first version without becoming a WordprocessingML nightmare.
